# Установочные скрипты

В этом репозитории живут **два разных установщика** для двух разных режимов
работы клиента. Не путай.

| Файл | Режим | Что делает | Используется визардом |
|---|---|---|---|
| **`install-projects.sh`** | **Mode 1 — Projects** | Поднимает наш бэкап-сервер для **двухсторонней** git-синхронизации проектов. **Без** монтирования хостовой ФС в контейнер. | «Создать сервер» |
| **`install-v2.sh`** | **Mode 2 — Prod** | То же, но **с монтированием** `/ → /host:ro` в контейнер, чтобы клиент мог делать **read-only зеркало** любых папок с проды (через external-проекты). | «Подключиться к проде» |
| `install.sh` | Mode 2 (legacy) | Старая версия Mode 2 без `--check`/`--apply` и без авто-апдейтов. Оставлен для обратной совместимости старых установок. | — (не используется визардами в новых клиентах) |

## Mode 1 (Projects) vs Mode 2 (Prod) — практическая разница

- **Mode 1** хранит на сервере **bare git-репозитории** проектов в
  `/opt/backups-app/data`. Клиенты пушат/пуллят свои локальные папки в эти
  репы. Сервер сигналит остальным через WebSocket. Двухсторонний sync.
  Хостовая файловая система контейнеру **не доступна** — всё, что есть, это
  репозитории, которые юзер сам залил.
- **Mode 2** монтирует **корень хоста** в контейнер read-only по адресу
  `/host:ro`. Это позволяет создавать «external project» с указанием пути на
  хосте (например, `/host/var/www/myapp`), и клиент будет тащить себе зеркало
  этой папки. **Односторонний** mirror, файлы локально только смотреть. На
  проде ничего не меняется.

## Идемпотентность и обновления

Оба новых скрипта (`install-projects.sh` и `install-v2.sh`):
- Поддерживают `--check` (только инспекция, печатает JSON состояния хоста) и
  `--apply` (ставит / обновляет до целевой версии).
- Сохраняют `.env` с админ-логином, паролем и JWT-секретом при повторном
  прогоне — апдейт не сбрасывает кредов.
- Печатают фазы (`BACKUPS_PHASE=...`) в stderr — клиент стримит их в UI.
- В конце печатают `BACKUPS_INSTALL_RESULT={...}` JSON в stdout — клиент
  парсит и сразу логинится.

## Ручной запуск

Mode 1 (свежий VPS, для проектов):
```bash
curl -fsSL https://raw.githubusercontent.com/Cotee4ka/backups-app/main/apps/server/scripts/install-projects.sh \
  | sudo bash -s -- --apply
```

Mode 2 (свежий VPS, для read-only зеркала с проды):
```bash
curl -fsSL https://raw.githubusercontent.com/Cotee4ka/backups-app/main/apps/server/scripts/install-v2.sh \
  | sudo bash -s -- --apply
```

После выполнения будет напечатан `pair-токен` (`bap2.<base64>...`), который
вставляется в клиент в соответствующем визарде.
